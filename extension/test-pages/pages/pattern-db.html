Pattern DB Test Page

Tests for pattern-based form detection using selector rules from the Pattern DB.

Expected Behavior
  • This page tests Pattern DB functionality
  • Pattern DB matches are evaluated before heuristic detection
  • When a pattern matches, its CSS selectors are used to find form fields
  • If no pattern matches, heuristic detection is used as fallback
  • The strict_subdomain flag affects credential matching

1. Selector-based Login Form

This form uses non-standard attributes that heuristic detection may not recognize. A pattern DB entry with explicit selectors would be needed to detect these fields.

Pattern DB entry for this form would use:
"username": { "selector": "#custom-user" }
"password": { "selector": "#custom-pass" }

2. Condition-based Split Login

Simulates a split login where username and password steps are determined by DOM conditions. Click the buttons to switch between steps.

Pattern DB would use:
"condition": { "element_exists": "#username-step:not([style*='display: none'])" }
to distinguish between steps.

3. Fallback Selectors

The primary selector targets an element that may not exist. The fallback selector catches the actual field.

Pattern DB entry would use:
"selector": "#primary-user", "fallback_selectors": [".alt-input[name='username']"]

4. Skip Fields

A form with an extra input that should be excluded from autofill via skip_fields.

Pattern DB entry would use:
"skip_fields": [".captcha-field"]

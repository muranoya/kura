# テスト方針

## 概要

vault-core（暗号化・同期・データ管理）に厚くテストを書き、UIは限定的にテストする。

パスワードマネージャーの正しさはセキュリティ・暗号化・同期ロジックに依存する。UIは比較的シンプルで変更頻度も高く、テストのROIが低い。

### vault-core

各モジュールにユニットテストを書く。VaultManager APIとsync APIのインテグレーションテストも行う。sync APIのテストにはMockStorageBackendを使用し、実際のストレージサービスには依存しない。

セキュリティクリティカルなコアライブラリであり、暗号化・復号・同期マージ・TOTP生成等のロジックの正しさを保証する必要がある。

### クライアントアプリ（デスクトップ・モバイル・拡張機能）

ロジック部分のみユニットテストを書く。vault-coreの薄いラッパー部分はテスト対象外（vault-core側でテスト済み）。

オンボーディングフロー（ウェルカム画面からストレージ設定を経てアイテム一覧画面に到達するまで）は分岐が多く手動テストが面倒なため、モックを使ったコンポーネントテストを書く。

## テストを書かない箇所

| 箇所 | 根拠 |
|------|------|
| vault-coreのAPIをそのまま呼ぶだけの薄いラッパー層 | vault-core側でテスト済みであり、二重テストになる |
| 外部SDKの薄いラッパー（StorageBackend実装等） | SDK自体のテストにしかならない |
| 単純なCRUD UI | 変更頻度が高くテストのメンテナンスコストが利益を上回る |
